## Overview of the Capstone Project

The MS Software Engineering (MSSE) Capstone Project gives you the opportunity to integrate and apply your knowledge of software engineering languages, tools, methods, and project management in producing, as part of a team, either:

a. **An AI system** (a software system built incorporating the use of AI models and using AI engineering techniques) or  
b. **Other software system**

**Duration:** Approximately three and a half months

**Focus Areas:**

- Software project management
- Software architecture and design
- Software testing
- AI-based applications and/or AI model fine-tuning
- Continuous integration/continuous delivery (CI/CD)
- Teamwork skills within a diverse, high-performing engineering team

**Key Philosophy:**

- Experiential approach: "learning by doing"
- Working cooperatively and constructively as part of a software or AI engineering team
- Developing real-world software or AI engineering capabilities

**Repository:**

- Software code will be hosted in a shared GitHub repository (with the option to make it public)
- Serves as a portfolio to showcase engineering skills to potential employers
- Approach with diligence and commitment to excellence

---

## Learning Outcomes

When completed successfully, this project will enable you to:

- Work as part of an agile engineering team to analyze, design, develop, test and deploy a high-quality software system, or AI application or agent
- Utilize the appropriate engineering methodology and tools for your project to deliver a software system on-time
- Demonstrate a working deliverable that meets agreed requirements to product stakeholders
- Demonstrate the use of appropriate AI tooling to support software development
- Utilize appropriate software and AI engineering techniques and technologies throughout

## Deliverables

The main deliverable of the MSSE Capstone is a **working, high-quality software system or AI system** meeting its user requirements.

While there is **no mandatory format** for the final deliverable, you must include:

### 1. Software Repository (GitHub)

- An accessible link to the GitHub repo containing the project's developed code
- Code should be appropriately documented
- **Must share repository with the GitHub account: `quantic-grader`**

### 2. Deployed Version Link

- The repository should include a link to the deployed version of the software or AI system (if a Web application)

### 3. Agile Task Board Link

- An accessible link to an agile task board (typically a Trello Scrum Board or other appropriate board)
- Documenting all tasks carried out and features delivered

### 4. Design and Testing Document

- Must be included in the repository
- Should list and explain:
  1. **Design and architecture decisions made**, including:
     - Any software and architectural patterns used
  2. **Details of software testing implemented**

### 5. Demonstration/Presentation Recording

- A link to a recording of a demonstration/presentation of the working software or AI system at the conclusion of the Capstone Project
- If a Web application: should be deployed to a free tier hosting environment such as Render, Railway, etc., or any other of your choice

## The Capstone Journey

The Capstone journey takes place over **three phases:**

1. Team formation and project conception
2. A series of agile development sprints
3. Final submission phase

---

### Team Formation and Project Conception

**Step 1: Assemble Team**

- Maximum of 6 group members
- Self-selected team members
- Look out for the signup sheet on Slack

**Step 2: Select Project Topic**
Project topics should be based on:

- Any team member can suggest a software application or AI use case from their current workplace or past work experience (simulated data can be used)
- Consider using approaches covered in the AI for Business specialization for AI use cases
- A hypothetical or novel application or product
- Projects that apply AI engineering techniques in a novel way to real-world use cases are encouraged—but not required

**Step 3: Product Owner Assignment**

- All team members must reach agreement on the specific project topic
- A group member will act as Product Owner for the software or AI system project
- If the project topic relates to a real-world workplace need known to a particular group member, that group member would be the Product Owner
- The Product Owner develops an initial list of user stories for the software or AI system

---

### Sprints

**Minimum Requirement:** At least three sprints by the end of the Capstone journey

#### First Meeting Agenda:

1. **Nominate and agree on a Scrum Master**
2. **Product Owner discusses:**
   - Scope of the project
   - Initial user stories with the team
3. **Develop a prioritized product backlog**
   - Product backlog items could consist of user stories, features, requirements, or other tasks
4. **Decide on collaborative tools and software technologies** to be used during the project
5. **Agree on sprint timeline** including the end date of the first sprint

#### First Sprint Planning Meeting:

- **First half:** Establish which user stories or other items from the product backlog will be delivered by the end of the first sprint (the sprint backlog)
- **Second half:** Team members along with the Scrum Master determine:
  - Individual tasks to be carried out
  - Who will do what
  - How to implement those agreed user stories or tasks during the sprint

#### During Each Sprint:

- **Maintain a Web-based Scrum Board** documenting:
  - Product backlog
  - User stories being implemented in each sprint
  - Specific constituent tasks
  - Completion status for each user story and task
- **Utilize CI/CD practices and tools** to collaboratively update, test, and develop the shared code base in the project's GitHub repository
- **At the end of each sprint:** Provide a recording of a software demonstration of working software to-date to the Product Owner for sprint review
- **Hold a sprint planning meeting** at the start of each new sprint and repeat for the next sprint cycle

**Deployment:** If the software or AI system is a Web application, by the time of submission, it should be deployed to a free tier production hosting environment such as Render, Railway, etc., or any other of your choice.

---

### Working in Teams

**Roles:**

- **Product Owner:** Contributes equally to software development work
- **Scrum Master:** Contributes equally to software development work
- **Team Members:** Including one or more Code Owners (approve pull requests to the GitHub repo)

**Team Dynamics:**

- Assign tasks optimally based on team members' particular skills
- Develop understanding and use of team dynamics
- Learn how to handle conflicts and resolve differences

**Regular Meetings:**

- Meet in regular time-boxed scrum meetings to discuss progress (approximately once a week)
- Attend and meaningfully participate in as many webinars and facilitated events offered as part of the Capstone journey as possible

---

### Sprint Planning Meetings

- **Attendance:** Must be attended by all team members
- **Purpose:** Plan for the coming sprint
- **Check-in Week:** In about a week, there will be a designated Check-in Week where you can schedule a check-in meeting with a Quantic faculty member to ask questions and receive feedback on work in progress

## Reporting and Submission

**Required Contents for Your Submission:**

### 1. Software Repository (GitHub)

- Link to an accessible software repository containing all developed code
- Code should be appropriately documented
- **Must share repository with the GitHub account: `quantic-grader`**

**The GitHub repository should include:**

- **Link to deployed version** of the software or AI system (if a Web application)
- **Link to accessible agile task board** (typically a Trello Scrum Board or other appropriate board) documenting:
  - All tasks carried out by group members
  - User stories delivered
- **Detailed design and testing document** listing and explaining:

  **i) Design and architecture decisions made:**
  - Technologies and architectural choices made and reasons for these
  - Any software and architectural patterns used and reasons used
  - Deployment options recommended for the software (e.g., on-premises or cloud)
  - Relative cost implications of the choice

  **ii) Software testing carried out:**
  - Document all testing done
  - Any and all automated tests used and reasons why

### 2. Recorded Demonstration/Presentation

- Link to a recorded demonstration/presentation of the working software or AI system
- Involves screen capture of the software being used with voice over
- **All group members must speak and be present on camera**
- **Presentation length:** Between 15 and 20 minutes

## Plagiarism Policy

**Definition:**
"Knowingly representing the work of others as one's own, engaging in any acts of plagiarism, or referencing the works of others without appropriate citation."

**Includes:**

- Misusing or not using proper citations for the works referenced (including any software code of others)
- Submitting someone else's work as your own

**Quantic's Approach:**

- Academic integrity is taken very seriously
- Monitors all submissions for instances of plagiarism
- All plagiarism, even unintentional, is considered a conduct violation
- **When in doubt, cite!**

**Code Citation:**

- [Guidance on appropriate citation of other programmer's code](https://quantic.edu/plagiarism-policy)

## Capstone Project Rubric

**Passing Scores:** 3 and above  
**Non-Passing Scores:** 0-2 (must revise and resubmit)

### Score 5 - Outstanding

**Addresses ALL of the Capstone requirements including, but not limited to:**

- Provides a software repository showing all developed project code. All code should be appropriately documented.
- Provides a link to the deployed version of the software or AI system (if a Web application).
- Provides an up-to-date task board showing completion of all agreed user stories and tasks
- Evidences well-designed and well-tested code via a detailed design and testing document. This should include any software and architecture patterns used and reasons for their use. It should also detail all testing done and testing methods used.
- Appropriate software engineering methodology and collaborative software engineering tools, including CI/CD tools, have been used
- **An outstanding recorded software demonstration** that clearly shows an outstanding software system.
- Shows initiative demonstrated by the project software deliverable **going above and beyond** the minimum Capstone requirements

### Score 4 - Excellent

**Addresses MOST of the Capstone requirements including, but not limited to:**

- Provides a software repository showing all developed project code. All code should be appropriately documented.
- Provides a link to the deployed version of the software or AI system (if a Web application).
- Provides an up-to-date task board showing completion of **most** agreed user stories and tasks
- Evidences **fairly** well-designed and **fairly** well-tested code via a detailed design and testing document. This should include any software and architecture patterns used and reasons for their use. It should also detail all testing done and testing methods used.
- Appropriate software engineering methodology and collaborative software engineering tools, including CI/CD tools, have been used
- **An excellent recorded software demonstration** that clearly shows an excellent software system.
- Shows good initiative

### Score 3 - Very Good (Minimum Passing)

**Addresses SOME of the Capstone requirements including but not limited to:**

- Provides a software repository showing all developed project code. **Most** code is appropriately documented.
- Provides a link to the deployed version of the software or AI system (if a Web application).
- Provides an up-to-date task board showing completion of **some** of the agreed user stories and tasks
- Evidences **moderately** well-designed and **moderately** well-tested code via a detailed design and testing document. The document should include any software and architecture patterns used and reasons for their use. It should also detail all testing done and testing methods used.
- Appropriate software engineering methodology and collaborative software engineering tools have been used
- **A very good recorded software demonstration** that clearly shows a very good software system.
- Shows some initiative

### Score 2 - Basic (Non-Passing)

**Addresses A FEW of the Capstone requirements including, but not limited to:**

- Provides a software repository showing all developed project code. **Little** of the code is appropriately documented.
- **Does NOT provide** a link to the deployed version of the software or AI system (if a Web application).
- Provides an up-to-date task board showing completion of a **few** agreed user stories and tasks
- Evidences **limited** design and testing of code via the design and testing document. Provides limited evidence of any software and architecture patterns used and reasons for their use. It provides limited information on testing done and testing methods used.
- Appropriate software engineering methodology and collaborative software engineering tools have been used, but with **very little use** of appropriate CI/CD tools.
- **A basic recorded software demonstration** that shows a good software system.
- Shows little initiative to go above and beyond the minimum requirements

### Score 1 - Poor (Non-Passing)

**Addresses the Capstone project requirements but MOST components are missing including, but not limited to:**

- Provides a software repository showing **limited** developed project code. The code is **not well documented**.
- **Does NOT provide** a link to the deployed version of the software or AI system (if a Web application).
- Provides an up-to-date task board showing completion of **very few** agreed user stories and tasks
- Evidences **poorly designed and tested** code via the design and testing document. Provides **no evidence** of any software and architecture patterns used. It provides **no information** on testing done and testing methods used.
- Appropriate software engineering methodology and collaborative software engineering tools have **largely not been used**, and **no use of CI/CD tools** made
- **No recorded software demonstration** is provided.
- Shows no initiative

### Score 0 - Incomplete (Non-Passing)

- The assignment is significantly incomplete, plagiarized, and/or most vital components are missing

## Capstone Presentation Rubric

**Passing Scores:** 2 and above  
**Non-Passing Scores:** 0-1 (must revise and resubmit)

### Score 5 - Outstanding

- Provides a **thorough, clear and concise** demonstration of the user stories implemented by the Capstone project software artifact or AI system for a **range of user inputs**
- The deliverable **operates correctly throughout** the demonstration
- The presentation is **professional** and the screen share is **clear and legible**
- The students are **clearly visible and audible** during the presentation
- The presentation is **between 15 and 20 minutes** long

### Score 4 - Excellent

- Provides a **thorough, clear and concise** demonstration of the user stories implemented by the Capstone project software artifact or AI systems
- The deliverable operates correctly throughout the demonstration with **only minor exceptions**
- The presentation is **professional** and the screen share is **clear and legible**
- The students are **clearly visible and audible** during the presentation
- The presentation is **marginally outside** of the time requirement

### Score 3 - Very Good (Minimum Passing)

- Provides a **clear and concise** demonstration of the **majority** of user stories implemented by the Capstone project software artifact or AI system
- The deliverable operates correctly throughout the demonstration with **only a few exceptions**
- The presentation is **professional** and the screen share is **clear and legible**
- The students are **clearly visible and audible** during the presentation
- The presentation is **significantly outside** of the time requirement

### Score 2 - Good (Minimum Passing)

- Provides a **mainly clear** demonstration of **some** of the user stories implemented by the Capstone project software artifact or AI system
- The deliverable operates correctly throughout the demonstration with **a number of exceptions**
- The presentation is **generally professional** and the screen share is **clear and legible**
- The students are clearly visible and audible during the presentation with **some exceptions**
- The presentation is **significantly outside** of the time requirement

### Score 1 - Poor (Non-Passing)

- Provides a **very limited** demonstration of **some** of the user stories implemented by the Capstone project software artifact or AI system
- The deliverable encounters **significant issues**/ does **not run correctly** during the demonstration
- The presentation is **unprofessional** and the screen share is **not clear and legible**
- The students are **not visible and/or audio is missing** during the presentation
- The presentation is **significantly outside** of the time requirement

### Score 0 - Incomplete (Non-Passing)

- The assignment is incomplete, plagiarized, and/or is missing most vital components
